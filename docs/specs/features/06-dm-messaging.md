# Slice 5 — DM Messaging (Write Path + Broadcast)

First messaging slice. Users exchange DMs in real-time via REST writes + SignalR broadcasts. Establishes the write-path pattern (REST POST → SignalR group broadcast) that slices 8+ extend to rooms.

## Context

`docs/implementation-plan.md` slice 5; depends on slice 3 (Friendship + PersonalChat entities exist) and slice 4 (ChatHub wired, `SignalrService` client-side).

Authoritative requirements:

- **Arch doc §DM design** — Messages created via `POST /api/chats/personal/{personalChatId}/messages` only if friendship is `accepted`. PersonalChat is the scope holder.
- **Arch doc §Message write path** — REST write → hub broadcast. Server resolves the SignalR group and calls `IHubContext<ChatHub>.Clients.Group(...)`.
- **Arch doc §Realtime topology** — `pchat:{personalChatId}` group; both participants receive `MessageCreated`.
- **Arch doc §Bounded contexts** — Messaging scope is cross-context; service consults Social for friendship/ban, Rooms for room member status (in slice 8). Single `ChatDbContext`.
- **Body size** — ≤ 3 KB per spec.

Outcome: A and B exchange DMs. A sends a message via REST → B's browser receives `MessageCreated` event immediately; both see the message in the chat view. Page refresh loads history.

## Decisions

| Topic | Decision | Rationale |
|---|---|---|
| Group join (server vs client) | `ChatHub.OnConnectedAsync` queries DB for user's personal chat IDs and joins each `pchat:{id}` group | No hub write methods; clean auth. New friends require reconnect to join their pchat group — acceptable for MVP. |
| Body validation | UTF-8 byte length ≤ 3000 (not char count) | Arch doc says 3 KB; multi-byte Unicode counted as bytes. Emoji and other multi-byte characters are accepted — the composer and server both measure the limit in UTF-8 bytes. |
| `Message.Scope` enum | `Personal = 0, Room = 1` | Room scope wired in slice 8; entity shape finalized now. |
| History order | Last 50 messages, `ORDER BY created_at ASC LIMIT 50` | Slice 15 adds keyset pagination. No pagination param yet. |
| `UnreadMarker` entity | Created; logic (increment on send) deferred to slice 9 | This slice's focus is write + broadcast, not unread. Entity shape must be correct now. |
| Reply-to | `reply_to_id` accepted, stored, not rendered | Rendering deferred to slice 10. Entity must support it. |

### Deferred (explicit — handed to later slices)

- Room scope (`POST /api/chats/room/{id}/messages`) → slice 8.

## Key flows (reference)

### Send a DM (happy path)

1. User A (browser 1) is in DM view with user B, types message, clicks Send.
2. `DmService.send(chatId, body)` → `POST /api/chats/personal/{chatId}/messages { body }`.
3. **Server:**
   - `PersonalMessagesController.Send()` validates auth (ICurrentUser).
   - `MessageService.SendAsync()` checks participant (404 if not), validates body ≤ 3000 bytes (400 if not).
   - Inserts `Message` row with `Scope = Personal`, `PersonalChatId = chatId`, `AuthorId = userId`, `Body = body`, `CreatedAt = UtcNow`.
   - `await db.SaveChangesAsync()`.
   - Resolves `pchat:{chatId}` group via `IHubContext<ChatHub>`. Broadcasts `MessageCreated` event with `MessageResponse` payload to the group.
   - Returns `201 Created` with `MessageResponse`.
4. **Hub delivery:**
   - Both A and B are in `pchat:{chatId}` group (B joined on their `ChatHub.OnConnectedAsync`; A joined on theirs).
   - `MessageCreated` event fires on both browser's `DmService` hub handler.
5. **Client A:** `DmService` appends to `messages` signal; Angular re-renders (no double-append because we await server response, not append optimistically).
6. **Client B:** `DmService` appends to `messages` signal; Angular re-renders. User sees A's message appear live.

### Load history

1. User navigates to DM view (`:chatId` param resolved).
2. `DmsComponent.ngOnInit()` calls `dmService.loadHistory(chatId)`.
3. `DmService` → `GET /api/chats/personal/{chatId}/messages`.
4. **Server:** `PersonalMessagesController.Get()` validates participant, queries last 50 messages ordered by `created_at ASC`, returns list.
5. **Client:** Results populate `messages` signal; template renders.

### Hub reconnect

1. API restarts. Both client connections close.
2. `SignalrService` reconnects both hubs per exponential backoff.
3. `ChatHub.OnConnectedAsync` fires: adds connection to `user:{userId}` group + re-joins all `pchat:{id}` groups (queried fresh from DB).
4. Chat view remains open; `messages` signal is stale but on next `MessageCreated` event it's correct.

## Implementation notes

### MessageService pattern

Follows `FriendshipService` pattern:
- Constructor injects dependencies (db, hub context).
- Methods return tuples `(bool Ok, string? Code, string? Message, T? Value)` for errors vs. success.
- Use `SocialErrors` / `MessagingErrors` constants for error codes.
- Queries use `AsNoTracking()` where safe (history loads).

### Message body validation

```csharp
var byteCount = Encoding.UTF8.GetByteCount(body);
if (byteCount > 3000)
    return (false, MessagingErrors.BodyTooLong, "Message exceeds 3 KB.", null);
```

### Group broadcast pattern

After persisting, obtain the hub context and call:
```csharp
await Clients.Group($"pchat:{personalChatId}")
    .SendAsync("MessageCreated", messageResponse, cancellationToken);
```

`IHubContext<ChatHub>` is injected into the service via constructor.

### UnreadMarker schema

Composite key `(user_id, scope, scope_id)` allows tracking unread count per unique (user, chat) tuple. Logic (incrementing on send, decrementing on mark-as-read) lands in slice 9.

## Verification

1. **Build clean.** `dotnet build server/ChatApp.sln` (zero warnings). `cd client && npm run build` (zero errors).
2. **Database.** `docker compose up -d --build`. Confirm `messages` and `unread_markers` tables exist in Postgres.
3. **Send DM (happy path).**
   - Two browsers, log in as User A and B.
   - Navigate A to contacts, click "Message" link for B.
   - A sends "Hello" → B's browser shows the message appear (no refresh).
   - Swap roles: B sends a message → A sees it live.
4. **History load.** Refresh DM view → existing messages load from `GET /api/chats/personal/{id}/messages`.
5. **Auth enforcement.** `curl -X GET /api/chats/personal/{id}/messages` without session cookie → `401`. (Or POST a new message.)
6. **Not-participant guard.** User C (not in the chat) tries `POST /api/chats/personal/{id}/messages` → `403 not_participant`.
7. **Body validation.** Send a message with > 3000 UTF-8 bytes → `400 body_too_long`.
8. **Hub group join.** API logs after `ChatHub connected` should mention group count, e.g., `… joined 3 pchat groups` (if user has 3 active personal chats). Can be a debug log.
9. **Unit tests** (in `ChatApp.Tests/Unit/Messaging/`):

   - `MessageService_SendAsync_NotParticipant_Returns403` — user not in the chat.
